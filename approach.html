Senior BI Delivery Sequence

This is the sequence I use for BI delivery: tight framing, governed metric ownership, disciplined model design, and controlled release so teams can trust and run the solution after handover.

Phase 1

Discovery & framing

Inputs (what I need)

  • Stakeholder goals, decision points, and current reporting pain points.
  • Source system inventory, data refresh expectations, and ownership contacts.

Outputs / artefacts

  • Agreed problem statement, delivery scope, and phase-level timeline.
  • Draft data-flow diagram showing source to semantic layer movement.

Common pitfalls + mitigation

  • Pitfall: Vague asks become moving targets. Mitigation: lock outcomes and non-goals in writing before build starts.
  • Pitfall: Assumed data availability. Mitigation: validate source fields and latency early with data owners.
Phase 2

KPI definition & stewardship

Inputs (what I need)

  • Business definitions for each KPI, including grain, owner, and action expected.
  • Policy or finance rules that influence numerator, denominator, and exclusions.

Outputs / artefacts

  • Baseline metric dictionary with formula logic, filter assumptions, and data lineage notes.
  • Simple KPI change-control log so definition updates are explicit and approved.

Common pitfalls + mitigation

  • Pitfall: Multiple versions of the same KPI. Mitigation: one owned definition and one semantic implementation.
  • Pitfall: KPI drift after release. Mitigation: route all logic changes through the change log and sign-off.
Phase 3

Data modelling & semantic layer design

Inputs (what I need)

  • Confirmed KPI logic, source relationships, and known dimensional hierarchies.
  • Constraints on refresh windows, security expectations, and data retention.

Outputs / artefacts

  • ERD for fact/dimension structure and semantic model boundaries.
  • Documented data-flow diagram and semantic layer naming conventions.

Common pitfalls + mitigation

  • Pitfall: Modelling for visuals instead of maintainability. Mitigation: enforce star-schema patterns and conformed dimensions.
  • Pitfall: Security retrofitted late. Mitigation: design and test row-level security with real user roles upfront.
Phase 4

Build & validation (reconciliation checks + controls)

Inputs (what I need)

  • Approved model structure and KPI definitions from earlier phases.
  • Control totals or trusted reports for baseline comparison.

Outputs / artefacts

  • Reconciliation checks between source, transformed, and semantic outputs.
  • Validation controls for refresh exceptions, threshold breaches, and issue triage.

Common pitfalls + mitigation

  • Pitfall: Reporting logic accepted without controls. Mitigation: publish reconciliation results with each release candidate.
  • Pitfall: Quiet refresh failures. Mitigation: implement alerting and escalation paths in the runbook.
Phase 5

Performance pass (semantic model + DAX Studio)

Inputs (what I need)

  • Representative user scenarios, filter paths, and high-cost visuals.
  • Baseline timings from model view and DAX Studio query traces.

Outputs / artefacts

  • Performance tuning notes covering model optimisations, measure rewrites, and trade-offs.
  • Updated semantic model checklist with accepted thresholds for refresh and query times.

Common pitfalls + mitigation

  • Pitfall: Tuning only once and too late. Mitigation: run performance passes before UAT and before production release.
  • Pitfall: Over-optimised complexity. Mitigation: keep measure patterns readable and document why any advanced pattern is used.
Phase 6

Release, handover & iteration

Inputs (what I need)

  • UAT feedback, production readiness checks, and stakeholder sign-off.
  • Support model, ownership map, and expected cadence for enhancement reviews.

Outputs / artefacts

  • Release notes, runbook, and handover pack with owners, SLAs, and rollback guidance.
  • Iteration backlog prioritised by business value and data quality risk.

Common pitfalls + mitigation

  • Pitfall: Knowledge trapped with the build team. Mitigation: walkthroughs and shared runbook ownership before go-live.
  • Pitfall: No post-release learning loop. Mitigation: schedule a 30-day review to tighten controls and prioritise next increments.
