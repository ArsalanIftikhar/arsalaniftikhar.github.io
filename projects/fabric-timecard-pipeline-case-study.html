Fabric Timecard Ingestion & Shift Reconstruction Pipeline

Built a Microsoft Fabric notebook pipeline to ingest workforce timecard/punch data via API, reconstruct shift-level clock-in/out and break records, and load curated outputs into a Lakehouse Delta table for downstream Power BI reporting.

Microsoft Fabric API Ingestion Python PySpark Delta Lake Power BI

Business need

Raw workforce punch data needed to be transformed into reporting-ready shift records for recurring workforce and operational reporting.

What changed

API-sourced punch events were reconstructed into curated clock-in/out, break and audit-ready shift-level records in a Fabric Lakehouse table.

Technical delivery

Implemented API ingestion, historical backfill, daily incremental loading and Delta MERGE/upsert logic in a Microsoft Fabric notebook.

Reporting value

Created a more reliable Lakehouse foundation for downstream Power BI reporting while reducing duplicate-load risk.

Business problem

Raw punch events from workforce systems are not directly reporting-ready. Clock-in, clock-out and break fields often need to be inferred from event sequences before analysts can build reliable operational reporting on top of the data.

Manual exports or ad hoc pulls can create repeatability issues, especially when reporting needs to refresh every day. A scheduled load can also create duplicate shift records if it blindly appends the same worker and shift combination after a rerun.

The reporting layer needed a curated table at the right grain: one row per reconstructed shift, with consistent start/end fields, break fields and load-audit information for downstream Power BI reporting.

Architecture / workflow

The workflow was designed as a practical Fabric notebook pattern that moves API-sourced timecard events into a curated Lakehouse output. API behaviour and request structure were investigated with Postman first, then implemented in an OAuth-aware notebook workflow using Python/PySpark transformations.

The solution sat within a medallion-style Fabric environment, with a lightweight Dev/Prod split to support safer release control for a small BI/data team.

Transformation logic

  • Punch events are sorted chronologically per worker_id.
  • Shift boundaries are inferred using a time-gap rule between consecutive events.
  • The first and last punch in a sequence are shaped into shift_start and shift_end.
  • Break events are mapped into break_start and break_end using sanitised event_type category logic.
  • The output is shaped into shift-level grain for reporting consumption.
  • A loaded_at audit timestamp supports traceability without exposing raw operational output.

Idempotent loading and Delta upsert

The notebook pattern supports historical backfill for initial loading and daily incremental mode for scheduled refresh. To reduce duplicate-load risk, the curated output is deduplicated around a composite business key such as worker plus shift start.

Delta MERGE/upsert logic prevents recurring loads from blindly appending duplicate shifts and makes reruns safer. This does not claim perfect data quality; it is a practical loading control that should be paired with validation checks for missing punches, unusual shift durations and other operational exceptions.

1. API call pattern

Generic OAuth-aware API request pattern for a single report date.

def fetch_timecard_data(report_date):
    token = get_access_token()
    payload = {
        "where": {
            "dateRange": {
                "startDate": report_date,
                "endDate": next_day(report_date)
            }
        },
        "select": ["PUNCH_EVENTS"]
    }

    response = requests.post(
        TIMEKEEPING_API_URL,
        headers={"Authorization": f"Bearer {token}"},
        json=payload,
        timeout=60
    )
    response.raise_for_status()
    return response.json()

2. Shift reconstruction logic

Generic sequence logic for reconstructing reporting-ready shifts from punch events.

def reconstruct_shifts(punch_events):
    shifts = []

    for worker_id, events in group_by_worker(punch_events):
        ordered = sorted(events, key=lambda x: x["punch_time"])
        current_shift = []

        for idx, event in enumerate(ordered):
            current_shift.append(event)

            is_last = idx == len(ordered) - 1
            starts_new_shift = (
                not is_last and
                ordered[idx + 1]["punch_time"] - event["punch_time"] > SHIFT_GAP_THRESHOLD
            )

            if is_last or starts_new_shift:
                shifts.append(build_shift_record(worker_id, current_shift))
                current_shift = []

    return shifts

3. Delta MERGE/upsert pattern

Generic Delta pattern for updating existing shift records and inserting new ones.

spark_df.createOrReplaceTempView("staged_timecard")

spark.sql("""
MERGE INTO curated_timecard AS target
USING staged_timecard AS source
ON target.worker_id = source.worker_id
AND target.shift_start = source.shift_start

WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *
""")
