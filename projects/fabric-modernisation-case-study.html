GXO Fabric Modernisation Case Study

Scoped contribution to dataset modernisation in Microsoft Fabric, focused on reliability, semantic clarity, and practical controls.

Context

Migration and modernisation of legacy reporting datasets in a live operational environment.

Role

Senior MI Analyst, contributing BI and data engineering components.

Tools / Stack

Microsoft Fabric, Lakehouse patterns, PySpark notebooks, SQL, Power BI semantic model, DAX Studio.

Ownership / Scope

Within my scope: ERD maintenance, data-flow documentation, reconciliation checks, DAX performance review, and runbook support.

Constraints

No disruption to operational reporting, evolving KPI definitions, and governance requirements on documentation and controls.

Problem

Reporting logic had developed across legacy assets over time, creating inconsistency in metric definitions and duplicated transformation logic. The modernisation workstream needed a reliable one-source-of-truth model so KPI definitions were consistent, duplicated report logic was reduced, and downstream reporting aligned to a governed semantic layer rather than report-by-report interpretation.

Because reporting requirements spanned multiple warehouses, the model also needed standardisation across sites: conformed entities and keys, consistent grain, and comparable reporting outputs regardless of source location.

Constraints

  • Operational stakeholders required uninterrupted reporting during transition phases.
  • KPI expectations changed as teams aligned on definitions across functions.
  • Documentation needed to be audit-friendly and suitable for handover.
  • Decisions on refresh and retention were made with wider team input; I supported these decisions within my remit.

My Approach

  • Contributed to migration activities by helping move legacy logic toward layered Lakehouse-aligned patterns.
  • Delivered within scope: ERD maintenance and data-flow diagrams to keep entity, key, and metric logic visible.
  • Supported standardisation across warehouses by implementing components such as conformed keys, consistent grains, and shared transformation rules for comparable reporting.
  • Partnered with stakeholders to align KPI interpretation and keep documentation current as logic evolved.
  • Supported ingestion and transformation work in PySpark notebooks and SQL for stable reporting inputs.

Grounded example: PySpark transform (anonymised)

This simplified pattern reflects the type of transformation component delivered within scope: standardise core fields, preserve latest records, and write audited silver outputs for governed semantic consumption.

from pyspark.sql import functions as F
from pyspark.sql.window import Window

bronze_df = spark.table("lakehouse.bronze_ops_events")

typed_df = (
    bronze_df
    .select(
        F.col("site_id").cast("string").alias("site_id"),
        F.to_date("event_date").alias("event_date"),
        F.col("entity_key").cast("string").alias("entity_key"),
        F.trim(F.upper(F.col("category"))).alias("category_std"),
        F.col("status").cast("string").alias("status"),
        F.col("source_updated_ts").cast("timestamp").alias("source_updated_ts")
    )
    .filter(
        F.col("site_id").isNotNull() &
        F.col("event_date").isNotNull() &
        F.col("entity_key").isNotNull()
    )
)

window_spec = Window.partitionBy("site_id", "event_date", "entity_key").orderBy(F.col("source_updated_ts").desc_nulls_last())

deduped_df = (
    typed_df
    .withColumn("rn", F.row_number().over(window_spec))
    .filter(F.col("rn") == 1)
    .drop("rn")
    .withColumn("record_hash", F.sha2(F.concat_ws("|", "site_id", "event_date", "entity_key", "category_std", "status"), 256))
    .withColumn("ingested_at_utc", F.current_timestamp())
    .withColumn("pipeline_run_id", F.lit("${run_id}"))
)

(deduped_df
    .write
    .mode("overwrite")
    .format("delta")
    .option("overwriteSchema", "true")
    .saveAsTable("lakehouse.silver_ops_events"))
