Microsoft Fabric migration notes: a practical guide from discovery to stable operations

This write-up summarises how I approach Fabric migration work in live BI environments: clarify purpose first, define architecture choices explicitly, implement controls early, and hand over with confidence.

What Microsoft Fabric is (and why teams use it)

Microsoft Fabric is an end-to-end analytics platform that brings ingestion, transformation, storage, modelling, and reporting capabilities into a single operating model. Teams use it to reduce platform fragmentation, simplify collaboration between data and BI roles, and build reusable delivery patterns instead of one-off report solutions.

  • One ecosystem for data engineering, analytical storage, semantic modelling, and consumption.
  • Better lineage visibility between source data, transformations, and business-facing outputs.
  • Improved consistency in naming, governance standards, and deployment practices.

Why organisations migrate to Fabric

  • Governance: centralised standards for ownership, lifecycle, and change control.
  • Standardisation: reduced variation in how teams build pipelines and semantic logic.
  • Less duplicated logic: fewer disconnected KPI definitions across notebooks, SQL, and reports.
  • Unified platform: smoother handoffs across engineering, analytics, and reporting teams.
  • Lineage and auditability: clearer traceability from source to dashboard output.
  • Scalable patterns: architecture that supports growth in volume, complexity, and usage.

Core architecture concepts and trade-offs

Lakehouse vs Warehouse: when to choose which

  • Lakehouse is usually preferred when ingestion sources are mixed, schema evolves, or transformation flexibility is important.
  • Warehouse is often better when teams need strongly modelled relational structures and SQL-first analytical consumption.
  • In practice, many delivery patterns use both: Lakehouse for processing layers and Warehouse/semantic outputs for governed consumption.

Medallion architecture (bronze / silver / gold)

  • Bronze: raw landed data with minimal transformation for traceability.
  • Silver: cleansed, conformed, and quality-checked entities.
  • Gold: business-ready structures and KPI outputs for semantic use.

Keeping these layers separate prevents business logic from being embedded too early and makes controls easier to run and explain.

PySpark role

PySpark is typically the right choice for scalable transformations, repeatable data engineering patterns, and workloads where row volumes or transformation complexity exceed lightweight tooling.

Dataflow Gen2 role (vs notebooks)

  • Dataflow Gen2 fits analyst-friendly shaping and smaller-to-medium complexity transformations.
  • Notebooks/pipelines are stronger for advanced orchestration, dependency management, and heavier processing.
  • Choosing intentionally avoids overlapping logic and unnecessary maintenance burden.

Refresh frequency and retention

  • Set refresh frequency based on decision cadence, not just technical capability.
  • Define retention windows with compliance and support needs in mind, not only storage cost.
  • Document archive/purge rules early to prevent uncontrolled data growth.

Partitioning considerations

  • Good partitioning improves performance and cost by limiting data scanned per query/refresh.
  • Common practical keys include date, site, region, and business unit depending on usage patterns.
  • Over-partitioning creates overhead, so tune iteratively based on actual workload behaviour.

Common pitfalls

  • Starting migration without stabilising KPI definitions and ownership.
  • Recreating duplicated logic in new tools instead of consolidating it.
  • Skipping reconciliation checks between source, transformed, and semantic layers.
  • Deferring performance review until late UAT or post-release firefighting.
  • Under-investing in runbooks and support documentation, increasing single points of failure.

My playbook (checklist)

  • □ Confirm business outcomes, KPI owners, and acceptable refresh windows.
  • □ Define medallion layer boundaries and naming/ownership conventions.
  • □ Choose Lakehouse/Warehouse usage based on workload characteristics.
  • □ Implement reconciliation controls as a release gate.
  • □ Complete a semantic model and DAX performance pass before production rollout.
  • □ Publish runbook, release notes, and escalation paths before handover.
