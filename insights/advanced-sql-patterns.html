Advanced SQL patterns for operational analytics

In operational BI, “advanced SQL” is less about clever syntax and more about query structures that preserve business meaning under scale: clear grain control, defensible joins, and checks that surface hidden rule breaks before they become KPI errors.

Pattern 1: lifecycle chaining (receipt → putaway) with CTEs

This pattern links two process stages while preserving event grain. The objective is to identify whether each receipt event was followed by a valid putaway event within an expected time window.

WITH receipt_events AS (
    SELECT
        r.site_id,
        r.product_id,
        r.unit_id,
        CAST(r.event_ts AS DATE) AS event_date,
        r.event_ts AS receipt_ts
    FROM ops_receipt_events r
    WHERE r.event_ts >= DATE '2025-01-01'
      AND r.event_ts < DATE '2025-02-01'
),
putaway_events AS (
    SELECT
        p.site_id,
        p.unit_id,
        p.location_code,
        p.event_ts AS putaway_ts
    FROM ops_putaway_events p
    WHERE p.event_ts >= DATE '2025-01-01'
      AND p.event_ts < DATE '2025-02-03'
),
chained AS (
    SELECT
        r.site_id,
        r.product_id,
        r.unit_id,
        r.event_date,
        r.receipt_ts,
        MIN(p.putaway_ts) AS first_putaway_ts
    FROM receipt_events r
    LEFT JOIN putaway_events p
      ON p.site_id = r.site_id
     AND p.unit_id = r.unit_id
     AND p.putaway_ts >= r.receipt_ts
    GROUP BY r.site_id, r.product_id, r.unit_id, r.event_date, r.receipt_ts
)
SELECT
    c.*,
    CASE
        WHEN c.first_putaway_ts IS NULL THEN 'missing_putaway'
        WHEN c.first_putaway_ts > c.receipt_ts + INTERVAL '24' HOUR THEN 'late_putaway'
        ELSE 'on_time'
    END AS lifecycle_status
FROM chained c;

Gotchas

  • Date filters: include a trailing buffer for downstream events so lifecycle chains are not falsely marked incomplete.
  • Duplicates: repeated scans can inflate joins; dedupe or aggregate intentionally at event grain.
  • Grain: define whether the lifecycle is tracked at unit, line, or document level before joining.
  • Null handling: preserve nulls from left joins so missing lifecycle stages stay visible for exception reporting.

Pattern 2: group-level rule validation (more than one distinct pallet)

This check finds assignment groups that break a business rule: each list should map to only one pallet in the active state.

SELECT a.site_id,
       a.assignment_list_id,
       COUNT(DISTINCT a.pallet_id) AS distinct_pallets
FROM ops_assignments a
WHERE a.status = 'ACTIVE'
  AND a.assignment_ts >= DATE '2025-01-01'
  AND EXISTS (
      SELECT 1
      FROM ops_assignments a2
      WHERE a2.site_id = a.site_id
        AND a2.assignment_list_id = a.assignment_list_id
        AND a2.status = 'ACTIVE'
      GROUP BY a2.site_id, a2.assignment_list_id
      HAVING COUNT(DISTINCT a2.pallet_id) > 1
  )
GROUP BY a.site_id, a.assignment_list_id
HAVING COUNT(DISTINCT a.pallet_id) > 1
ORDER BY distinct_pallets DESC;

Performance notes

  • Indexes: prioritise composite indexes on `(site_id, assignment_list_id, status, assignment_ts)` for selective scans.
  • EXISTS vs JOIN: use `EXISTS` for rule checks when you only need presence, avoiding row multiplication from joins.
  • Sargable filters: keep date predicates on raw columns (no wrapping functions) so index seeks remain viable.

Mini-pattern 3: window function for latest valid event

Window functions are essential for selecting canonical records without losing audit traceability.

WITH ranked_events AS (
    SELECT
        e.site_id,
        e.entity_key,
        e.event_ts,
        e.event_status,
        ROW_NUMBER() OVER (
            PARTITION BY e.site_id, e.entity_key
            ORDER BY e.event_ts DESC
        ) AS rn
    FROM ops_entity_events e
    WHERE e.event_ts >= DATE '2025-01-01'
)
SELECT
    site_id,
    entity_key,
    event_ts AS latest_event_ts,
    event_status
FROM ranked_events
WHERE rn = 1;

This pattern is commonly paired with lifecycle chaining to stabilise “latest state” reporting in semantic models.
